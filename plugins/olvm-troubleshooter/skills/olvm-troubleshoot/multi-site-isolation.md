# 멀티사이트 회사 작업 격리 원칙

## 왜 이게 중요한가

같은 회사의 여러 사이트는 다음 이유로 작업 사고 위험이 높음:

- 호스트명 패턴 유사 (seoul-host01, busan-host01, daejeon-host01)
- 같은 AD/LDAP 도메인 인증으로 여러 사이트 접속 가능
- 같은 명령어 규약/디렉토리 구조
- 운영팀이 같으면 머릿속에 여러 사이트 정보 혼재

**세션 혼동 사고**: busan에서 작업한다고 생각하며 seoul에 명령 전송 → VM 다운 등 실 사고 발생 가능.

## Claude가 지켜야 할 원칙

### 원칙 1: 항상 사이트 코드를 prefix로 명시

조치 명령어 안내 시 반드시 사이트/호스트 코드 표기.

```
❌ "systemctl restart vdsmd 실행해주세요"
❌ "host01에 SSH 접속해서..."

✅ "[seoul-dc01/host03] systemctl restart vdsmd 실행해주세요"
✅ "[seoul-dc01] host03 (10.10.1.13, FQDN seoul-host03.local) 에서 다음 명령..."
```

### 원칙 2: SSH 접속 안내는 풀 정보로

호스트명만 안내하면 다른 사이트로 잘못 갈 수 있음. 항상:
- 사이트 코드
- 호스트 FQDN 또는 IP
- 경유 Bastion 정보

```
❌ "host01에 접속"
✅ "[seoul-dc01] seoul-host01.local (10.10.1.11), bastion-seoul.local 경유"
```

### 원칙 3: 사용자 응답에서 사이트 검증

사용자가 명령 결과를 붙여넣을 때, 결과에 나타난 호스트명/IP/MAC이 현재 작업 중인 사이트와 일치하는지 확인.

불일치 발견 시 즉시 멈추고 확인:

```
⚠️ 사이트 불일치 의심

현재 작업 중인 사이트: seoul-dc01
방금 받은 결과의 호스트명: busan-host02 (다른 사이트로 보임)

다음 중 하나일 것 같습니다:
1. 사이트가 바뀌었나요? 그렇다면 /olvm-troubleshoot busan-dc01 으로 다시 시작해주세요.
2. 잘못 붙여넣으신 건가요? 그렇다면 seoul-dc01 호스트 결과를 다시 보내주세요.
3. 같은 회사의 다른 사이트 호스트인데 우연히 출력된 건가요? 확인 부탁드립니다.

이대로는 안전하게 진행하기 어려워서 멈춰두겠습니다.
```

### 원칙 4: 다중 사이트 동시 작업 금지

한 번에 한 사이트만 트러블슈팅. 회사 차원의 작업(예: 라이선스 갱신, AD 변경)이 필요하면 별도로 분리해서 안내.

사용자가 "busan도 같은 문제인 것 같으니 같이 보자"고 하면:
- 현재 사이트 작업 완료 후 별도 세션
- 또는 명확히 단계 분리: "1단계: seoul-dc01 처리 → 완료 → 2단계: busan-dc01 처리"

### 원칙 5: 회사 차원 영향 명시

조치 전 다음 명시:
- 이 조치의 영향 범위: 사이트 한정 vs 회사 전체
- 공유 인프라(AD/모니터링/백업/DR) 영향 여부

예시:
```
🟡 [seoul-dc01] LDAP 인증 설정 변경

⚠️ 회사 차원 영향 가능
이 회사는 사내 AD (corp.local)를 seoul-dc01 / busan-dc01 / daejeon-dc01 모두 공유합니다.
LDAP 설정 변경은 OLVM 측만 영향 (Engine 설정).
AD 자체는 변경 X.
다른 사이트 OLVM 인증에 영향 없음.

만약 AD 자체 변경이 필요한 거라면 별도 회사 차원 변경 관리 절차 필요.
```

## 위험 시나리오 예시

### 시나리오 1: AD 계정 잠금
- 상황: seoul 사이트에서 admin 비밀번호 여러 번 시도 → 잠금
- 영향: 같은 AD 쓰는 busan, daejeon 사이트도 잠김
- 가드: 비밀번호 시도 전 AD 공유 여부 확인. 잠금 시 즉시 AD 관리자 연락.

### 시나리오 2: SSH known_hosts 충돌
- 상황: seoul-host01과 busan-host01이 같은 호스트명만 다른 IP
- 영향: known_hosts 충돌. 호스트 키 검증 우회하면 MITM 위험.
- 가드: 항상 FQDN 또는 IP로 접속. 호스트 키 우회 (`-o StrictHostKeyChecking=no`) 금지.

### 시나리오 3: DNS 캐시 / 환경 변수
- 상황: 사용자 노트북에 seoul-dc01 환경변수 남은 상태로 busan-dc01 작업
- 영향: DNS lookup 결과가 잘못된 사이트로 향함
- 가드: 작업 시작 시 `nslookup <대상호스트>` 로 IP 확인 후 진행.

### 시나리오 4: 백업 스크립트 충돌
- 상황: 회사 차원 백업 스크립트가 모든 사이트 동시 실행
- 영향: 한 사이트 점검 중 자동 백업 시작 → 부하 증가
- 가드: 점검 전 백업 스케줄 일시 정지 또는 시간 조정.

### 시나리오 5: 모니터링 알람 폭증
- 상황: 점검 중 호스트 maintenance 전환 → 통합 모니터링이 다른 사이트에도 알람 전송
- 영향: 운영팀 혼란
- 가드: 점검 전 모니터링 silence 설정 (대상 호스트만 한정).

## 작업 시작 체크리스트 (Claude 항상 수행)

트러블슈팅 시작 시:

1. **사이트 식별 확정**: `[{company}/{site}]` 표기
2. **회사 _company.md 의 사이트 간 관계 확인**:
   - 공유 인프라가 무엇인가
   - DR 관계가 있는가
   - 작업이 다른 사이트에 영향 줄 수 있는가
3. **사용자에게 명시**: "이 트러블슈팅은 {site} 한정입니다. 다른 사이트({related_sites})에는 영향 없습니다."
4. **공유 인프라가 있으면 추가 경고**: "단, AD 인증을 모든 사이트가 공유하므로 LDAP 설정 변경 시 영향 가능."

## 비상시 (실수 발생 시)

사용자가 "잘못된 사이트에 명령 실행했다"고 신고:

1. 즉시 영향 평가:
   - 어떤 명령이었는가
   - 어느 호스트/VM에 영향
   - 데이터 변경 발생했는가
2. 위험도별 대응:
   - 조회 명령 → 영향 없음, 안심 안내
   - 서비스 재기동 → 해당 사이트 상태 확인, 정상이면 통보
   - 데이터 변경 (drop/delete/destroy) → L3/PM 즉시 보고, 백업 복원 검토
3. 사고 기록:
   - 잘못 작업한 사이트의 `.md` 작업 이력에 기록
   - 의도한 사이트의 `.md` 알려진 이슈에 "오작업으로 영향 받았음" 기록
4. 재발 방지: 호스트 식별 가드를 더 보수적으로 적용 권고.

## 사용자 환경 가이드

운영팀에 권장:

- 셸 프롬프트에 현재 SSH 접속한 호스트명 표시
- 다른 사이트로 접속 시 색상 변경 (PS1 커스터마이즈)
- `~/.ssh/config` 에서 사이트별 Host 별칭 명시 (seoul-host01 vs busan-host01 혼동 방지)
- tmux session 이름에 사이트 코드 포함 (`tmux new -s seoul-dc01`)
- 명령 실행 전 `whoami; hostname; hostname -f` 확인 습관화
