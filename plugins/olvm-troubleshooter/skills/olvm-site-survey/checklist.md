# 사이트 등록 9단계 체크리스트

각 단계의 질문 문구, 받아야 할 정보, frontmatter 매핑, 본문 위치를 정의.

## 회사 인터뷰 (회사 _company.md 신규 생성 시)

### Q-C1. 회사 정식 명칭
- 질문: "회사 정식 명칭이 어떻게 되나요?"
- 저장: frontmatter `company_name`, 본문 H1
- 미입력 허용: 아니오 (필수)

### Q-C2. 계약 형태
- 질문: "계약 형태는? (a) 유지보수 (b) 기술지원 (c) 구축"
- 저장: frontmatter `contract_type` (maintenance / tech_support / build)
- 미입력 허용: 예

### Q-C3. SLA
- 질문: "SLA 응답 시간(분), 해결 시간(시간) 알려주세요. 모르면 기본값 30분/4시간 사용."
- 저장: frontmatter `sla_response_minutes`, `sla_resolution_hours`

### Q-C4. 주요 담당자
- 질문: "회사측 OLVM 총괄 담당자와 우리측 계정 PM 이름/연락처 알려주세요. 비밀번호는 받지 않습니다."
- 저장: 본문 "담당자/연락처" 섹션
- 비밀번호/계정 비밀번호 입력 시: 거부하고 "별도 보안 저장소에 두세요" 안내

### Q-C5. 사이트 개수와 코드 목록
- 질문: "이 회사는 사이트가 몇 개인가요? 각 사이트 코드(소문자 하이픈 권장)도 알려주세요."
- 저장: frontmatter `sites_count`, 본문 "사이트 인벤토리" 표 초기 행 생성
- 답변 예: "3개 — seoul-dc01, busan-dc01, daejeon-dc01"

### Q-C6. 사이트 간 관계
- 질문: "사이트 간 DR 관계 / 데이터 복제 / 공유 인프라가 있나요? 없으면 '없음'."
- 검증: 공유 인프라(AD/모니터링/백업) 있으면 작업 시 영향 범위 주의 강조
- 저장: 본문 "사이트 간 관계" 섹션

### Q-C7. 에스컬레이션 체인
- 질문: "L1→L2→L3→임원 보고 기준이 회사 표준과 다른가요? 같으면 기본값 사용."
- 저장: 본문 "에스컬레이션 체인" 섹션

### Q-C8. 변경 관리 절차
- 질문: "CAB 승인 필요 항목과 변경 윈도우 알려주세요."
- 저장: 본문 "변경 관리 절차" 섹션

## 사이트 인터뷰 (사이트 .md 채우기)

### 단계 1: 기본 정보

#### Q1-1. 사이트 명칭과 위치
- 질문: "이 사이트의 정식 명칭과 위치(주소 또는 데이터센터명) 알려주세요."
- 저장: frontmatter `site_name`, `location`, 본문 H1과 기본 정보

#### Q1-2. 지역 (region)
- 질문: "지역 분류는? (예: 수도권, 영남, 호남 등) 자유 입력."
- 저장: frontmatter `region`

#### Q1-3. 역할
- 질문: "이 사이트의 역할은? (a) primary 운영 (b) secondary (c) DR (d) 개발/테스트"
- 저장: frontmatter `role` (primary / secondary / dr / dev)

#### Q1-4. 독립도
- 질문: "다른 사이트와의 독립 정도는? (a) 완전 독립 (b) 스토리지 공유 (c) 네트워크 공유 (d) 인증 공유"
- 저장: frontmatter `independence`
- 검증: 'fully_independent' 아니면 작업 시 영향 범위 안내 필요

#### Q1-5. 운영 시간 / 점검 윈도우
- 질문: "이 사이트의 운영 시간(예: 24/7)과 정기 점검 윈도우(예: 매월 둘째 주 토 02-06) 알려주세요."
- 저장: 본문 "작업 시간대 제약" 섹션

#### Q1-6. 현장 출입 절차
- 질문: "현장 출입 절차/사전 통보 시간 있으면 알려주세요. 없으면 '없음'."
- 저장: 본문 "사이트 기본 정보" 섹션

---

### 단계 2: OLVM 환경

#### Q2-1. OLVM 버전
- 질문: "OLVM 버전 알려주세요. 모르면 'rpm -q ovirt-engine' 결과 붙여주세요."
- 저장: frontmatter `olvm_version`

#### Q2-2. Oracle Linux 버전
- 질문: "Oracle Linux 버전은? (예: 8.8) 모르면 'cat /etc/oracle-release' 결과."
- 저장: frontmatter `ol_version`

#### Q2-3. Engine 형태
- 질문: "Engine은 standalone 별도 서버인가요, SHE(Self-Hosted Engine)인가요?"
- 저장: frontmatter `she_enabled` (true/false), 본문 OLVM 환경 섹션
- 후속: SHE면 → Q2-3a, standalone이면 → Q2-3b

##### Q2-3a. SHE 호스트 수 (SHE인 경우)
- 질문: "SHE 호스트가 몇 대인가요? (Engine VM이 떠 있을 수 있는 호스트)"
- 저장: frontmatter `she_host_count`
- 검증: 1대만이면 HA 안 됨 → 위험 표시. 2대 이상 권장.

##### Q2-3b. Standalone Engine IP (standalone인 경우)
- 질문: "Engine 서버 IP/호스트명 알려주세요."
- 저장: 본문 OLVM 환경 섹션

#### Q2-4. 콘솔 URL
- 질문: "OLVM 콘솔 URL은? (예: https://olvm.example.local/ovirt-engine)"
- 저장: 본문 OLVM 환경 + 인증 섹션

#### Q2-5. Data Center / Cluster 이름
- 질문: "OLVM 콘솔에서 사용하는 Data Center 이름과 Cluster 이름은?"
- 저장: 본문 OLVM 환경 섹션

#### Q2-6. CPU Type
- 질문: "클러스터 CPU Type 설정은? (예: Intel Cascadelake Family)"
- 저장: 본문 OLVM 환경 섹션
- 검증: 'Lowest Common Denominator'로 다운그레이드되어 있으면 마이그레이션 호환은 좋지만 성능 손실 표시

### 단계 3: 호스트 인벤토리

#### Q3-1. 입력 방식 선택
- 질문: "호스트 정보를 (a) 한 대씩 묻고 답하기 (b) 표 한꺼번에 붙여넣기 중 어느 게 편하세요?"
- 표 붙여넣기 예: `호스트명 | IP | iLO | 역할 | CPU | RAM`

#### Q3-2. 호스트 수
- 질문: "총 호스트 몇 대인가요?"
- 저장: frontmatter `host_count`

#### Q3-3. 호스트별 (각 호스트 반복)
- 호스트명 / 관리 IP / iLO IP / 역할(SHE/일반/SPM 우선) / CPU 모델 / RAM / 디스크 / 비고
- 저장: 본문 "호스트 인벤토리" 표
- 검증:
  - SHE 표시한 호스트 수가 Q2-3a 답과 일치하는가
  - 호스트마다 같은 CPU 패밀리인가 (다르면 마이그레이션 영향 표시)

#### Q4-2. VM망

#### Q4-5. 본딩

## 종료 처리

모든 단계 완료 또는 사용자 중단 시:

1. frontmatter `last_updated` 갱신
2. `pending_fields` 배열 갱신 (빈 값으로 둔 필드 키 목록)
3. risk_level 자동 평가:
   - SHE 호스트 1대만 → high
   - 스토리지망 미분리 → high
   - 백업 복구 테스트 3개월 이상 → medium 이상
   - 호스트 CPU 패밀리 혼재 → medium 이상
4. 회사 `_company.md` 의 사이트 인벤토리 표 갱신
5. 요약 출력: "채워진 항목 N / 미확인 M / 자동 평가 risk_level: X"
